1. What Milestone 3 currently does
Accepts a user question from CLI input loop.
Runs hybrid retrieval:
Graph retrieval from Neo4j using keyword-extracted Cypher queries.
Semantic retrieval from Pinecone vector store.
Builds one merged context block:
Graph facts section
Email snippets section
Sends merged context + question to LLM endpoint for grounded answer generation.
Prints answer in terminal.
2. How answer generation works in Milestone 3
Entry point is generate_answer(question) in MIlestone-3.py.
It calls retrieve_hybrid(question), which internally combines:
retrieve_graph(question)
retrieve_vector(question)
The final prompt enforces strict grounding and citation style from provided context only.
LLM call is made through HTTP POST to configured chat completions endpoint.
Returned text is the final answer shown to user.
3. How results are shown today
Current output target is terminal only.
For each query, script prints:
Query marker
Generated answer text
Divider line
There is no HTTP API yet, so frontend cannot consume responses directly until a thin service wrapper is added.
4. Graph representation available for frontend
Neo4j stores entities and dynamic relationships (not fixed to a tiny hardcoded relation list).
Current retrieval returns graph facts as strings in this shape:
Source [RELATION] Target
For frontend graph visualization, transform each fact string into:
Node source
Node target
Edge relation label
Optional edge weight if frequency exists in relation text
Recommended response should include both:
Raw graph facts for audit/evidence display
Normalized nodes and edges for graph canvas rendering
5. What frontend should expect from backend per query
Return one combined payload containing:

User question
Final generated answer
Graph facts list
Email snippets list
Graph object:
nodes: id, label, type if known
edges: source, target, relation, weight optional
Optional diagnostics:
retrieval counts
latency per stage
warnings or partial-failure messages
6. Minimal backend changes needed to connect frontend
Keep all retrieval and generation functions unchanged in MIlestone-3.py.
Add an API layer that calls existing functions instead of CLI loop.
Replace terminal loop with service endpoints, for example:
health endpoint
query endpoint
Query endpoint should:
validate question
call retrieval functions
call generate_answer
return structured JSON payload
Preserve current grounding behavior and deterministic settings.
7. Required runtime dependencies for frontend integration
Neo4j connectivity and credentials.
Pinecone index availability and matching embedding settings.
LLM API key and endpoint configured exactly as backend expects.
All required environment variables loaded before service starts.
8. Known operational characteristics frontend should handle
Query latency can vary due to multi-service retrieval + generation.
Partial retrieval failure is possible:
Graph retrieval may fail while vector retrieval still works, or vice versa.
Frontend should render:
answer even when one evidence source is missing
warning banners when partial context used
Empty-evidence cases should be shown gracefully.
